# Docker Architecture Explained

Docker follows a **Client-Server Architecture** where the Docker Client sends commands to the Docker Engine, which manages Docker resources such as Images, Containers, Networks, and Volumes.

## 1. Docker Client

This is where users interact with Docker using commands.

Examples:

```bash
docker build
docker pull
docker run
docker ps
docker stop
```

The Docker Client sends these commands to the Docker Engine.

---

## 2. Docker Engine

Docker Engine is the core component of Docker that processes commands and manages containers.

It consists of:

### Docker Daemon (dockerd)

The background service responsible for:

- Building images
- Creating containers
- Managing networks
- Managing volumes

### REST API

Provides an interface through which Docker Client communicates with Docker Engine.

### Docker CLI

Command-line tool used by users to interact with Docker.

---

## 3. Docker Registry

A Docker Registry stores Docker Images.

The Docker Engine can:

- Pull images from a registry
- Push images to a registry

Examples:

- Docker Hub
- Amazon ECR
- Google Artifact Registry

Example:

```bash
docker pull nginx
docker push myimage
```

---

## 4. Docker Objects

Docker Engine manages four main objects:

### Images

Read-only templates used to create containers.

Example:

```bash
docker pull ubuntu
```


### Containers

Running instances of Docker Images.

Example:

```bash
docker run ubuntu
```


### Networks

Allow communication between containers.

Example:

```bash
docker network create mynetwork
```

### Volumes

Provide persistent storage for containers.

Example:

```bash
docker volume create myvolume
```

---

# Complete Workflow

```text
User
  │
  ▼
Docker Client
  │
  ▼
Docker Engine
  │
  ├── Pull/Push Images ↔ Docker Registry
  │
  ├── Create Images
  ├── Create Containers
  ├── Create Networks
  └── Create Volumes
```

## Example

- If we run:    ```
                docker run nginx
              ```

Step 1: Docker Client sends the command.

Step 2: Docker Engine receives the request.

Step 3: Docker Engine checks whether the nginx image exists locally.

Step 4: If not available, it pulls the image from Docker Hub.

Step 5: Docker Engine creates a container from the image.

Step 6: The container starts and the application begins running.

- This entire process happens automatically through Docker Architecture.

# Docker Lifecycle

- The Docker Lifecycle represents the stages an application goes through, from creating a Docker Image to running and managing a Docker Container.

## Docker Lifecycle Stages

## Step 1: Create Dockerfile

A Dockerfile contains instructions required to build a Docker Image.

Example:

```dockerfile
FROM python:3.12-slim
COPY . /app
CMD ["python3","app.py"]
```

Purpose:

- Defines application environment
- Specifies dependencies
- Provides startup commands

---

## Step 2: Build Image

Docker reads the Dockerfile and creates a Docker Image.

Command:

```bash
docker build -t myapp .
```

Purpose:

- Packages application code
- Packages dependencies
- Creates a reusable image

---

## Step 3: Docker Image

A Docker Image is a read-only template used to create containers.

Command:

```bash
docker images
```

Purpose:

- Stores application code
- Stores libraries and dependencies
- Acts as a blueprint for containers

---

## Step 4: Run Container

A container is created from the image.

Command:

```bash
docker run -d --name mycontainer myapp
```

Purpose:

- Creates a running environment
- Starts the application

---

## Step 5: Running Container

The application is now running inside the container.

Command:

```bash
docker ps
```

Purpose:

- View running containers
- Monitor application status

---

## Manage Container State

A container can move between different states.

### Stop Container

```bash
docker stop mycontainer
```

The container stops executing but still exists.

### Start Container

```bash
docker start mycontainer
```

Starts a previously stopped container.

### Restart Container

```bash
docker restart mycontainer
```

Stops and starts the container again.

---

## Step 6: Remove Container

Delete a container when it is no longer needed.

Command:

```bash
docker rm mycontainer
```

Purpose:

- Free resources
- Clean up unused containers

---

## Step 7: Remove Image

Delete the Docker Image.

Command:

```bash
docker rmi myapp
```

Purpose:

- Remove unused images
- Reclaim storage space

### Key Concept

- **Docker Image** = Blueprint
- **Docker Container** = Running instance of the Image

One Docker Image can create multiple Docker Containers.

## Key Point

- A Docker Image is a blueprint, while a Docker Container is a running instance of that image.
